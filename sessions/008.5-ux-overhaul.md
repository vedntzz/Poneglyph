## Retro

### What worked

- **Bottom-up build order was correct.** All 6 components built in isolation, build verified clean, then wired into the page. No compile errors during assembly.
- **Pre-populating documents before the run starts.** The spec called for documents showing `pending` status before the pipeline runs. This makes the page look alive even before clicking Run — the PM sees 5 documents waiting to be processed.
- **Ref pattern for SSE handlers.** Caught the stale closure issue before it manifested as a runtime bug. EventSource captures the handler once — without refs, late-pipeline events (verified_section, done) would see stale evidenceByImage state.
- **Keeping MemoryFeed as a collapsible debug view.** Engineers who want to see the plumbing can open `<details>` — PMs never see it. Both audiences served without two separate pages.

### What didn't work

- **No full pipeline visual test in this session.** The demo takes 7–9 minutes of real API calls. We verified the build, SSE endpoint, and static assets, but didn't run the complete pipeline. Items 7–10 in acceptance criteria are "wired" not "visually verified."
- **Drift Timeline data depends on `done` event containing contradictions.** The backend's `done` event may not include contradiction data in the payload — need to verify the backend emits it. If not, the drift timeline will be empty even after a successful run.
- **Document status tracking is heuristic.** Matching Scout filenames to document IDs via `FILENAME_TO_DOC_ID` works for the canonical demo, but relies on the backend emitting filenames that exactly match. If the backend sends full paths or different casing, the matching breaks.

### What to change

- **Session 010 should start with a full E2E browser run** (accept the 8-minute wait) to verify all motion treatments and choreography visually.
- **Check backend `done` event payload** — does it include contradiction data? If not, add a final contradiction summary to the orchestrator's done event.
- **Consider adding a mock SSE replay mode** for rapid UI iteration without burning API tokens. Record one full pipeline's events, replay them at 10x speed.
